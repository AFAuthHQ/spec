# AFAuth v0.1 — Implementation Scope & Roadmap

> Bridges the normative spec (`spec/core.md`) to the v0.1 reference implementation.
> Versioned with the spec, not the SDK. When the spec advances to v0.2, this
> document is superseded by `v0.2-scope.md`.

## Goal

A working reference implementation of AFAuth v0.1 that lets:

- a developer stand up an AFAuth-protected service on Cloudflare Workers in under an hour;
- an agent author sign and call AFAuth endpoints in roughly ten lines of TypeScript;
- a third-party implementer measure conformance against the shipped vectors.

The bar is **reference quality**, not production. Production polish (observability,
rate-limiting, alerting, audit pipelines) is explicitly deferred.

## Non-goals (v0.1)

- Recipient handlers beyond `email`. The other three types (`phone`, `oidc`, `did`)
  are interface-only in the SDK; the reference Worker does not ship implementations.
- `did:web` resolution. Only `did:key` in v0.1, in-process.
- Account/session storage. The SDK defines `AccountStore`; the reference Worker
  uses Cloudflare KV with a minimal schema.
- Refresh tokens, long-lived agent sessions, or any token-based auth layer.
  AFAuth v0.1 signs every request; token issuance is out of scope and tracked
  separately as a proposal.
- RFC 9421 component coverage beyond `@method`, `@target-uri`, `content-digest`.
  This matches what `vectors/signatures/` exercises today.
- Browser SDK. The agent SDK is Node 18+ only in v0.1.
- Performance benchmarks. Useful, but a separate workstream.

## Components

| Artefact | Repo path | Purpose |
|---|---|---|
| `@afauthhq/core` | `AFAuthHQ/typescript-sdk` — `packages/core/` | Shared primitives: `did:key` codec, RFC 9421 canonicalisation, content-digest, error envelope, types |
| `@afauthhq/agent` | `AFAuthHQ/typescript-sdk` — `packages/agent/` | Agent SDK: `Agent` class, signing, protocol-aware request builders, discovery client |
| `@afauthhq/server` | `AFAuthHQ/typescript-sdk` — `packages/server/` | Service SDK: `Verifier`, `Server`, `NonceStore`, `AccountStore`, `RecipientHandler` |
| `@afauthhq/worker` | `AFAuthHQ/typescript-sdk` — `packages/worker/` | Cloudflare bindings: `createWorker`, `KvNonceStore`, `WorkerOptions` |
| `afauth-worker-template` | `AFAuthHQ/typescript-sdk` — `examples/worker/` | Reference Cloudflare Worker that composes `@afauthhq/server` + `@afauthhq/worker` |
| `afauth` (CLI) | `AFAuthHQ/cli` | Go binary (cobra) — keygen, sign, verify, discover. Reimplements Ed25519 + did:key + RFC 9421 natively rather than calling the TS SDK |

The SDK ships as four packages under a pnpm workspace. `@afauthhq/core` is a
shared dependency of the other three so the canonicalisation rule, codec,
and error envelope stay aligned across the SDK. Consumers install only what
they need: an agent author depends on `@afauthhq/agent`; a service depends on
`@afauthhq/server` and optionally `@afauthhq/worker`.

The CLI is intentionally a separate codebase in Go. It distributes as a
single static binary (no Node runtime required) and verifies that the
protocol primitives are language-independent — a useful cross-check on the
spec's wire format.

## Architecture & API decisions

All decisions are recorded as ADRs in [`adr/`](adr/) — refer there for full
context, alternatives considered, and consequences. Headline summary:

1. **Nonce store: Cloudflare KV with TTL** ([ADR-0001](adr/0001-kv-nonce-store.md)).
   Eventual consistency is acceptable because replay risk is bounded by the
   signature's `expires` parameter. The `NonceStore` interface is pluggable;
   Durable Objects can replace KV later without an API change.

2. **Framework: raw Workers, no router library** ([ADR-0002](adr/0002-raw-workers-no-framework.md)).
   The Worker has five routes; a router framework would dominate bundle
   size and pull in dependencies the SDK does not need.

3. **DID resolver: `did:key` only, in-process** ([ADR-0003](adr/0003-did-key-only-resolver.md)).
   A resolver hook is exposed for future `did:web` and other methods;
   only `did:key` ships in v0.1.

4. **SDK API shape** ([ADR-0004](adr/0004-sdk-api-shape.md)). `AccountStore`
   exposes named atomic operations (not generic CRUD) so storage-layer
   atomicity is the implementation's contract. `Server.handleClaimCompletion`
   takes an explicit `session` parameter; the Worker layer accepts an
   `extractOwnerSession` callback for uniform routing. `Agent.signRequest`
   stays public with spec-conformant defaults.

## Implementation milestones

Each milestone is gated by **objective conformance signal**: the named harness
vectors must pass with the new code paths exercised end-to-end, not just unit-tested.

| ID | Scope | Conformance gate |
|---|---|---|
| **M0** | Repos scaffolded, CI green on placeholder suite, the four SDK packages published to reserve names on npm, reference Worker boots and returns 200 on `/.well-known/afauth` | none (skeleton) |
| **M1** | Agent SDK signs requests; Worker exposes discovery; `Verifier` accepts harness vectors | `vectors/signatures/*` (all 8) pass through SDK's `Verifier.verify()` |
| **M2** | Owner-invitation (email only), claim completion, two-step verify enforced; reference email handler logs the magic link | manual end-to-end: `afauth-cli` initiates invite, claim completes |
| **M3** | Pre-claim key rotation, owner-initiated revocation, revocation list as KV | regression test: post-rotation signature with old key returns `401 revoked_key` |
| **M4** | Full error envelope (§11) codes; replay detection via nonce store; clock-skew tolerance | C.5 (when written), C.6 (when written) |
| **M5** | `afauth` CLI v0.1: `init`, `whoami`, `call <url>`, `signup <service-url>`, `invite <recipient>`, `accounts list/show`, `keys rotate/export/import`, plus `discover <url>` (added in M5) | Worker conformance is gated on the SDK test suite + Appendix C vectors in CI |

Milestones M2 and M3 each need at least one new test vector pinned to the
behaviour they unlock; those vectors land in the same PR as the implementation.

## Test strategy

- **Unit:** canonicalisation, did:key codec, content-digest computation, error
  envelope shape, signature param parsing.
- **Conformance:** every committed vector in `vectors/signatures/` must pass
  through the SDK's `Verifier.verify()` (not the harness's standalone path).
  This is the contract between the SDK and the spec.
- **Integration:** a wrangler-deployed Worker serves the discovery doc and the
  five endpoints; `@afauthhq/server`'s test suite runs the Appendix C vectors
  against it in CI.
- **Property:** light fuzzing of canonicalisation against the harness's
  byte-exact expectations.

We are **not** writing exhaustive snapshot tests for HTTP response bodies.
The vectors are the source of truth for canonical input; response shapes are
covered by integration tests.

## Out of scope, tracked separately

- did:web resolver
- Multi-key accounts (key fan-out)
- Audit log spec and webhook delivery shape
- Threat model addendum (separate doc, M2-aligned)
- Browser SDK
- Production deployment guide

## Status

M0–M5 are complete with CI green on both reference implementations:

- `AFAuthHQ/typescript-sdk` — the four `@afauthhq/*` packages plus
  `examples/worker`.
- `AFAuthHQ/cli` — the `afauth` Go binary (clean-room second
  implementation of the wire primitives).

Conformance gates met for v0.1:

- §C.1 / §C.2 — all 8 signature vectors verify through both
  `Verifier.verify` (TypeScript) and the Go `proto.BuildCanonicalInput`
  + `crypto/ed25519.Verify` path. Byte-exact cross-language match.
- §C.3 — all 16 discovery-document vectors accept or reject in both
  TS and Go validators.
- §C.4 — all 19 recipient-normalisation vectors round-trip through
  the TS and Go normalisers with byte-identical canonical output.
- §C.5 — all 17 §11.3 error codes have envelope fixtures (one per
  reserved code, including `owner_session_too_stale` from AFAP-0002).
- §C.6 — all 4 replay-window vectors (expired, future-dated,
  fresh+replay, cross-keyid) produce the expected outcome.
